# Implementation Plan: Multi-Phase App Creation with Auto-Merge Pipeline Orchestration

**Branch**: `copilot/create-implementation-plan-for-app` | **Date**: 2026-04-06 | **Spec**: Boykai/solune#998
**Input**: Parent issue #998 — Multi-Phase App Creation with Auto-Merge Pipeline Orchestration

## Summary

When a user creates a new app, Solune will: (1) generate an initial plan via the built-in chat plan agent, (2) launch `speckit.plan` as a Copilot coding agent to produce a detailed `plan.md`, (3) parse the phases from `plan.md`, (4) create one GitHub Parent Issue per phase with Agent Pipelines, (5) execute phases sequentially — each phase auto-merges to main before the next starts — so subsequent phases always branch from updated code.

The technical approach reuses existing infrastructure: `assign_copilot_to_issue()` for agent assignment, `execute_pipeline_launch()` for pipeline creation, queue mode with a new `prerequisite_issues` field for wave-based execution, and `auto_merge=True` per pipeline for automatic merge-then-dequeue flow.

## Technical Context

**Language/Version**: Python >=3.12 (backend), TypeScript 5.x (frontend)
**Primary Dependencies**: FastAPI, Starlette (BackgroundTask), aiosqlite, React 18, Copilot SDK
**Storage**: SQLite (existing — new `app_plan_orchestrations` table, metadata JSON for `prerequisite_issues`)
**Testing**: pytest (backend), Vitest (frontend)
**Target Platform**: Linux server (self-hosted GitHub App), Web browser (frontend)
**Project Type**: Web application (backend + frontend)
**Performance Goals**: Plan orchestration completes within 20 minutes; status polling < 100ms; WebSocket updates < 500ms latency
**Constraints**: speckit.plan agent takes 5–15 minutes (async background task required); single-instance deployment (no distributed locking needed)
**Scale/Scope**: Single concurrent orchestration per app; up to ~10 phases per plan

## Implementation Phases

### Phase 1 — Plan Agent Integration (Backend Service Layer)

**Depends on**: Nothing (foundation phase)

**Step 1.1**: Create `app_plan_orchestrator.py` service

New file `solune/backend/src/services/app_plan_orchestrator.py` with class `AppPlanOrchestrator`:

- `orchestrate_app_creation(app_name, description, project_id, pipeline_id, session)` — main orchestration method:
  1. Update status → `planning`
  2. Call `chat_agent_svc.run_plan(message=description)` to get structured plan summary
  3. Update status → `speckit_running`
  4. Create temporary "planning issue" via `github_service.create_issue()`
  5. Assign `speckit.plan` via `copilot_service.assign_copilot_to_issue(custom_agent="speckit.plan", custom_instructions=plan_summary)`
  6. Poll for "Done!" marker via `copilot_service.check_agent_completion_comment()` (30s intervals, 20min timeout)
  7. Discover PR branch from the planning issue
  8. Fetch `plan.md` via `github_service.get_file_content_from_ref(path="specs/.../plan.md", ref=pr_branch)`
  9. Close the temporary planning issue
  10. Update status → `parsing_phases`
  11. Parse plan.md → `PlanPhase` objects via `plan_parser.parse_plan()`
  12. Update status → `creating_issues`
  13. Create phase issues (Step 1.3)
  14. Update status → `launching_pipelines`
  15. Launch pipelines with wave-based execution (Phase 2)
  16. Update status → `active`

- `_update_status(orchestration_id, new_status)` — updates DB record and broadcasts WebSocket event
- `_create_phase_issues(phases, app_name, description, project_id, pipeline_id, session)` — creates GitHub issues per phase

**Step 1.2**: Create `plan_parser.py`

New file `solune/backend/src/services/plan_parser.py`:

- `parse_plan(plan_md_content: str) -> list[PlanPhase]`:
  1. Find `## Implementation Phases` section
  2. Extract `### Phase N — Title` blocks via regex `r"### Phase (\d+)\s*[—–-]\s*(.+)"`
  3. For each phase block, extract:

     - Description: first paragraph after heading
     - Steps: numbered lines (`1.`, `2.`, etc.) or `**Step N.N**` patterns
     - Dependencies: scan for "depends on Phase N" → `depends_on_phases`
     - Execution mode: "parallel with Phase N" → `execution_mode = "parallel"`
  4. Validate: phase indices are sequential, dependencies reference valid phases
  5. Return `list[PlanPhase]`

- `group_into_waves(phases: list[PlanPhase]) -> list[list[PlanPhase]]`:
  1. Wave 1: phases with no dependencies (`depends_on_phases == []`)
  2. Wave N: phases whose dependencies are all in Waves 1..N-1
  3. Return list of waves (each wave is a list of phases)

- `@dataclass PlanPhase`: `index`, `title`, `description`, `steps`, `depends_on_phases`, `execution_mode`

**Step 1.3**: Create phase issue builder in `AppPlanOrchestrator`

For each `PlanPhase`, create a GitHub Parent Issue:

- Title: `Phase {N}/{M}: {phase.title} — {app.display_name}`
- Body contains:
  - App overview: "This is Phase {N} of {M} in building {app.display_name}: {description}"
  - Phase content from plan.md (steps, dependencies)
  - Agent tracking table via `append_tracking_to_body()`
- Add to project board at "Backlog" status
- Store issue numbers in `AppPlanOrchestration.phase_issue_numbers`

### Phase 2 — Sequential Phase Execution with Auto-Merge

**Depends on**: Phase 1

**Step 2.1**: Add `prerequisite_issues` field to `PipelineState`

In `solune/backend/src/services/workflow_orchestrator/models.py`:

- Add field: `prerequisite_issues: list[int] = field(default_factory=list)`
- Place after the `queued` field

In `solune/backend/src/services/pipeline_state_store.py`:

- `_pipeline_state_to_row()`: Add `prerequisite_issues` to metadata dict
- `_row_to_pipeline_state()`: Read `prerequisite_issues` from metadata dict with default `[]`

**Step 2.2**: Extend `execute_pipeline_launch()` with new params

In `solune/backend/src/api/pipelines.py`:

- Add parameters: `auto_merge: bool = False`, `prerequisite_issues: list[int] | None = None`
- Pass `auto_merge=auto_merge` to `PipelineState()` constructor
- Pass `prerequisite_issues=prerequisite_issues or []` to `PipelineState()` constructor
- Both queue and non-queue code paths must set these fields

**Step 2.3**: Launch phases with wave-based queuing

In `AppPlanOrchestrator._launch_phase_pipelines()`:

1. Call `plan_parser.group_into_waves(phases)` to get wave ordering
2. For Wave 1: call `execute_pipeline_launch()` for each phase with `auto_merge=True`, `prerequisite_issues=[]`
3. For Wave 2+: call `execute_pipeline_launch()` for each phase with `auto_merge=True`, `prerequisite_issues=[issue_numbers_from_prior_waves]`, and the pipeline will be queued automatically (queue mode)
4. All pipelines in the same wave can run in parallel (if queue mode allows)

**Step 2.4**: Extend `_dequeue_next_pipeline()` with prerequisite checking

In `solune/backend/src/services/copilot_polling/pipeline.py`:

- After fetching the next queued pipeline, before marking as active:
  1. Check `next_pipeline.prerequisite_issues`
  2. If non-empty, verify all prerequisite issues have their main PR merged
  3. Use `github_service.get_issue()` or check pipeline state for "Done" status
  4. If any prerequisite is unmet, skip this pipeline and try the next queued one
  5. If all prerequisites met, proceed with dequeue as normal

**Step 2.5**: Verify auto-merge → dequeue trigger (no code change needed)

Confirmed: `pipeline.py` already calls `_dequeue_next_pipeline()` after successful auto-merge at line ~2641. The trigger reason `auto_merge_done(issue=#{issue_number})` is logged. No changes needed.

## Key Decisions

| Decision | Rationale | Alternatives Rejected |
|----------|-----------|----------------------|
| Two-stage planning (chat agent → speckit.plan) | Chat agent provides fast structured seed; speckit.plan produces detailed plan.md | Single-stage: less structured input for speckit.plan |
| Queue mode + prerequisite_issues | Reuses existing queue infrastructure; lightweight extension | Full DAG scheduler: over-engineered for sequential phases |
| auto_merge=True per pipeline | Explicit per-phase control; more precise than project-level toggle | Project-level auto_merge: affects unrelated pipelines |
| Background orchestration | speckit.plan takes 5–15 min; API returns immediately | Synchronous: HTTP timeout risk; polling-only: misses real-time updates |
| Metadata JSON for prerequisite_issues | No schema migration needed; fits existing serialization pattern | New SQL column: unnecessary migration for transient field |
| WebSocket for progress | Established pattern in Solune for build/pipeline progress | SSE: used for chat but not pipeline progress; polling: wasteful |
