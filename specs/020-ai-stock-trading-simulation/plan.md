# Implementation Plan: AI Stock Trading Simulation App

**Branch**: `copilot/speckit-plan-actionable-steps` | **Date**: 2026-04-08 | **Spec**: [#1116](https://github.com/Boykai/solune/issues/1116)
**Input**: Parent issue #1116 — AI Stock Trading Simulation App — Azure OpenAI + Microsoft Agent Framework

## Summary

Build a full-stack AI-powered stock trading simulation application that uses Azure OpenAI models and Microsoft AutoGen (Agent Framework) to simulate intelligent trading decisions. The app features a React dashboard (Tailwind v4 + Vite), a FastAPI backend (aiosqlite), and a multi-agent AI layer where specialized agents (MarketAnalystAgent, RiskManagerAgent, TraderAgent) collaborate to research stocks, evaluate signals, manage risk, and execute simulated trades. The simulation uses synthetic or delayed stock price data — no real money is exchanged. All agent conversations and decisions are logged for audit and replay.

The stack follows the existing Solune monorepo conventions: `solune/frontend` (React + Tailwind v4 + Vite), `solune/backend` (FastAPI + aiosqlite), containerized via Docker Compose, with Azure OpenAI providing the LLM backbone and Microsoft AutoGen orchestrating the agent pipelines.

## Technical Context

**Language/Version**: Python >=3.12 (backend), TypeScript 6.0 (frontend)
**Primary Dependencies**: FastAPI, aiosqlite, openai SDK, pyautogen / autogen-agentchat, yfinance (optional), React 19, Tailwind v4, recharts, @tanstack/react-query, Vite
**Storage**: SQLite via aiosqlite (existing pattern — WAL mode); new tables: `portfolios`, `holdings`, `orders`, `trades`, `agent_sessions`, `agent_messages`, `simulations`
**Testing**: pytest (backend), Vitest + Testing Library (frontend)
**Target Platform**: Linux server (Docker Compose), Modern browsers (frontend at localhost:5173)
**Project Type**: Web application (backend + frontend monorepo under `solune/`)
**Performance Goals**: WebSocket price ticks <500ms latency; agent conversation streaming <1s per turn; dashboard renders at 60fps; API endpoints <200ms p95 response
**Constraints**: Fully offline-capable (synthetic data fallback); no real financial transactions; Azure OpenAI endpoint required for agent orchestration (graceful degradation when unavailable); simulation-only disclaimer on all UI surfaces
**Scale/Scope**: Single-user simulation; up to 50 watchlist symbols; up to 10 concurrent agent sessions; portfolio history up to 10,000 trades

## Design Decisions

| Decision | Rationale | Alternatives Rejected |
|----------|-----------|----------------------|
| Geometric Brownian motion for synthetic data | Produces realistic OHLCV patterns for offline simulation; no external API dependency | Pure random walk (unrealistic volatility), hardcoded JSON (no variability) |
| yfinance for delayed real data (optional) | Free, well-maintained Python library for Yahoo Finance delayed quotes | Alpha Vantage (API key required, rate limits), Polygon.io (paid), IEX Cloud (paid) |
| AutoGen GroupChat for multi-agent | Microsoft-supported framework; GroupChat pattern matches analyst→risk→trader workflow | LangChain agents (more complex, less structured), CrewAI (less mature), custom implementation (reinventing the wheel) |
| SQLite (existing) for all persistence | Consistent with Solune conventions; WAL mode handles concurrent reads; no new infrastructure | PostgreSQL (overkill for single-user sim), Redis (volatile), in-memory (no persistence) |
| recharts for P&L charting | Lightweight, React-native, declarative API; sufficient for line/bar charts | D3 (too low-level), Chart.js (wrapper needed), TradingView widget (heavy, licensing) |
| FIFO cost basis for P&L | Standard accounting method; simplest correct implementation | LIFO (less common), average cost (hides individual trade performance) |
| WebSocket for real-time feeds | Existing pattern in Solune (websocket.py); bidirectional, low latency | SSE (one-directional, insufficient for market feeds), polling (high latency, wasteful) |
| Separate API router files per domain | Follows existing Solune convention (apps.py, agents.py, board.py) | Single monolithic router (unmaintainable), nested routers (over-engineered) |

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| AutoGen multi-agent framework | Core requirement: 3 specialized agents must collaborate with structured conversation flow | Single LLM call (no inter-agent reasoning), sequential function calls (no collaboration) |
