# Feature Specification: Performance Review — Balanced First-Pass Optimization

**Feature Branch**: `003-performance-review`  
**Created**: 2026-04-05  
**Status**: Draft  
**Input**: User description: "Perform a balanced first pass focused on measurable, low-risk performance gains across backend and frontend"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Baseline Performance Measurement (Priority: P1)

A team member opens the performance review workflow and, before any optimization work begins, captures concrete performance baselines for both the backend and frontend. They record how many external API calls the backend makes while a board sits idle over a fixed interval, the time and cost of a single board data request, and how the real-time and polling refresh paths behave. On the frontend they profile board load time, interaction responsiveness, and identify the top rerender sources. These baselines are documented in a before/after checklist so that every subsequent optimization can be measured against a known starting point.

**Why this priority**: All other work depends on having reliable baselines. Without measurement, optimizations are assumptions — they cannot be validated, and regressions cannot be detected. This story blocks every optimization story.

**Independent Test**: Can be fully tested by instrumenting the system and recording metrics over a controlled interval. Delivers a documented baseline checklist that the team can reference throughout the project.

**Acceptance Scenarios**:

1. **Given** an open board with representative data, **When** the board sits idle for five minutes, **Then** the number of backend API calls to external services is recorded and documented as the idle-activity baseline
2. **Given** a board endpoint request, **When** the request completes, **Then** the request duration and the count of downstream external API calls are recorded as the per-request cost baseline
3. **Given** the frontend board is loaded with a representative project, **When** the initial load completes, **Then** the time-to-interactive and total component render count are recorded as the frontend load baseline
4. **Given** the real-time sync and fallback polling paths are active, **When** no data has changed on the backend, **Then** the network activity generated by each refresh path over a five-minute window is recorded
5. **Given** baseline measurements are collected, **When** the before/after checklist is populated, **Then** every metric has a numeric value, a measurement method, and a target improvement range

---

### User Story 2 - Backend Idle API Consumption Reduction (Priority: P1)

A user opens a project board and leaves it idle. Previously, the system continued making repeated external API calls even when nothing changed. After optimization, the backend detects when data is unchanged and suppresses redundant refresh calls. The real-time subscription path only triggers a board data refresh when it receives a genuine change signal. Sub-issue data is served from a warm cache during board refreshes instead of re-fetching every time. Fallback polling does not accidentally trigger expensive full-board refreshes.

**Why this priority**: Reducing idle API consumption is the highest-value backend fix. It directly lowers rate-limit risk, reduces latency for genuine refreshes, and prevents the system from wasting resources on unchanged data. This is the primary backend optimization target.

**Independent Test**: Can be tested by opening a board, leaving it idle for a measured interval, and comparing the external API call count against the baseline. A passing test shows a material reduction (target: 50% or greater reduction in idle API calls).

**Acceptance Scenarios**:

1. **Given** a board is open and idle with no data changes, **When** the real-time subscription receives a refresh signal, **Then** the backend checks whether board data has actually changed before triggering a downstream API call
2. **Given** sub-issue data was fetched within the cache validity window, **When** a board refresh occurs, **Then** sub-issue data is served from the cache without additional external API calls
3. **Given** the primary real-time connection is unavailable and fallback polling is active, **When** a polling cycle completes, **Then** only lightweight status checks are performed — not full board data refreshes
4. **Given** a user manually triggers a board refresh, **When** the refresh request is processed, **Then** caches are bypassed and fresh data is fetched from external services
5. **Given** the backend optimizations are applied, **When** idle API activity is measured over the same interval as the baseline, **Then** external API call count is reduced by at least 50% compared to the baseline

---

### User Story 3 - Frontend Refresh Path Decoupling (Priority: P2)

A user is viewing a board when a lightweight task update arrives (e.g., a status change on a single issue). Previously, any update — regardless of scope — would invalidate the entire board data query, causing a full re-fetch and re-render. After optimization, lightweight updates are applied incrementally without triggering the expensive board data query. Only a manual refresh or a structural board change (column added/removed, bulk update) triggers a full board re-fetch. The user experiences faster, smoother updates for common operations.

**Why this priority**: Frontend refresh path decoupling prevents the "polling storm" pattern where small updates cascade into expensive full-board refreshes. This directly improves perceived responsiveness and reduces unnecessary network traffic.

**Independent Test**: Can be tested by sending a single-task update through the real-time channel and verifying that the board data query is not invalidated. Then performing a manual refresh and verifying the full query does fire.

**Acceptance Scenarios**:

1. **Given** a board is displayed and a single-task status change arrives via real-time sync, **When** the update is processed, **Then** only the affected task is updated in the UI without triggering a full board data query invalidation
2. **Given** fallback polling is active, **When** a polling cycle detects no structural board changes, **Then** lightweight task updates are applied without invalidating the board data query
3. **Given** the user clicks the manual refresh button, **When** the refresh is processed, **Then** the full board data query is triggered and caches are bypassed
4. **Given** a structural board change occurs (column added or removed), **When** the change is detected, **Then** a full board data query is triggered to reflect the structural change
5. **Given** the frontend refresh optimizations are applied, **When** a series of single-task updates arrive, **Then** no redundant board queries appear in the network activity log

---

### User Story 4 - Frontend Board Render Optimization (Priority: P2)

A user interacts with a board that has many columns and issues. Previously, every update — even to a single card — would cause the entire board to re-render, including all columns and all cards. After optimization, only the affected components re-render. Derived data (sorting, aggregation) is computed once and cached rather than recalculated on every render. Heavy event listeners (drag handlers, popover positioning) are throttled so they do not fire more often than necessary. The user experiences noticeably smoother board interactions, especially on larger boards.

**Why this priority**: Render optimization targets the most visible user-facing performance issue. Users directly perceive sluggish board interactions, and the fixes here are low-risk (memoization, stable props, throttling) without requiring new dependencies or architectural changes.

**Independent Test**: Can be tested by profiling board interaction (drag, scroll, card click) before and after optimization. A passing test shows a measurable reduction in unnecessary re-renders and improved interaction responsiveness.

**Acceptance Scenarios**:

1. **Given** a board with multiple columns and many cards, **When** a single card's data changes, **Then** only that card and its parent column re-render — other columns and cards remain stable
2. **Given** a page component computes derived data (sorting, grouping) from board data, **When** the board data has not changed, **Then** the derived data is served from a memoized cache rather than recomputed
3. **Given** a user is dragging a card across the board, **When** the drag event fires, **Then** event handlers execute at a throttled rate (no more frequently than every 16ms) to maintain smooth visual feedback
4. **Given** a popover or tooltip is open on a board card, **When** the popover position is recalculated, **Then** the recalculation is throttled to avoid excessive layout work
5. **Given** the render optimizations are applied, **When** frontend render metrics are compared to the baseline, **Then** unnecessary re-renders during common board interactions are reduced by at least 40%

### User Story 5 - Verification and Regression Coverage (Priority: P3)

A developer completes the optimization work and runs the verification suite. Backend tests confirm that cache behavior, change detection, and polling logic work correctly. Frontend tests confirm that refresh decoupling and board hook behavior are correct. A manual pass confirms that real-time updates still arrive promptly, fallback polling is safe, manual refresh bypasses caches, and board interactions feel responsive. The before/after checklist is completed with post-optimization measurements demonstrating the improvements are real.

**Why this priority**: Verification is essential to prove that optimizations are genuine improvements and not regressions in disguise. Without this story, the team cannot confidently ship the changes.

**Independent Test**: Can be tested by running the full verification suite (automated tests plus one manual network/profile pass) and comparing post-optimization measurements against the baseline checklist.

**Acceptance Scenarios**:

1. **Given** backend optimization changes are complete, **When** the backend test suite targeting cache, board, projects, and polling behavior is run, **Then** all tests pass with no regressions
2. **Given** frontend optimization changes are complete, **When** the frontend test suite targeting real-time sync and board refresh hooks is run, **Then** all tests pass with no regressions
3. **Given** all automated tests pass, **When** a manual end-to-end check is performed, **Then** real-time updates arrive within 2 seconds, fallback polling does not trigger full refreshes, and manual refresh bypasses caches
4. **Given** the before/after checklist exists with baseline values, **When** post-optimization measurements are recorded, **Then** every metric shows improvement or no regression, and the checklist is complete with both baseline and final values

---

### User Story 6 - Second-Wave Readiness Assessment (Priority: P3)

After the first-pass optimizations are verified, a team member reviews the results to determine whether deeper structural changes are needed. If large boards still exhibit noticeable lag after the low-risk frontend fixes, the recommended next step is board virtualization. If backend API consumption remains high after the targeted fixes, the recommended next step is deeper service consolidation. The outcome of this assessment is a documented recommendation — either "first pass is sufficient" or a scoped follow-on plan.

**Why this priority**: This story ensures the team has a clear exit criterion and a path forward if the first pass falls short. It prevents open-ended optimization without measurable decision points.

**Independent Test**: Can be tested by reviewing the completed before/after checklist against the success criteria. If targets are met, the assessment concludes with "sufficient." If targets are missed, a scoped follow-on plan is produced.

**Acceptance Scenarios**:

1. **Given** the first-pass optimizations are verified and the before/after checklist is complete, **When** all success criteria are met, **Then** the assessment concludes with "first pass is sufficient" and no follow-on work is planned
2. **Given** the before/after checklist shows large boards still have noticeable interaction lag after frontend fixes, **When** the assessment is performed, **Then** the recommendation is to pursue board virtualization as the next step
3. **Given** the before/after checklist shows backend API consumption remains above target after fixes, **When** the assessment is performed, **Then** the recommendation is to pursue deeper service consolidation as the next step

---

### Edge Cases

- What happens when the real-time connection drops and fallback polling activates during an optimization measurement window? Measurements should be re-taken under stable connection conditions, or the fallback scenario should be measured separately.
- How does the system behave when a board has zero issues (empty board)? Optimizations should not introduce errors or unnecessary processing on empty boards.
- What happens when multiple users have the same board open and one triggers a manual refresh? The manual refresh should only affect the requesting user's cache; other users' idle behavior should remain optimized.
- What if the cache validity window expires exactly during a board refresh? The system should handle cache expiration gracefully, falling back to a fresh fetch without errors.
- What happens when a board has an unusually large number of columns or issues (e.g., 50+ columns, 500+ cards)? The optimizations should degrade gracefully — performance should improve, not worsen, even if the improvement is smaller on very large boards.
- What if the real-time channel delivers updates out of order? The frontend should apply updates idempotently so that out-of-order delivery does not cause visual inconsistencies.

## Requirements *(mandatory)*

### Functional Requirements

**Phase 1 — Baseline and Guardrails**

- **FR-001**: System MUST provide a mechanism to measure and record the number of external API calls made during a fixed idle interval for an open board
- **FR-002**: System MUST provide a mechanism to measure and record the per-request cost (duration and downstream call count) of a board data endpoint
- **FR-003**: System MUST provide a mechanism to profile and record frontend board load time, time-to-interactive, and component render counts
- **FR-004**: System MUST produce a before/after checklist document with baseline values for all measured metrics before any optimization code changes are made

**Phase 2 — Backend API Consumption Fixes**

- **FR-005**: System MUST detect when board data is unchanged and suppress redundant external API calls during real-time subscription refresh processing
- **FR-006**: System MUST serve sub-issue data from a warm cache during board refreshes when the cache is within its validity window
- **FR-007**: System MUST ensure that fallback polling performs only lightweight status checks and does not trigger full board data refreshes
- **FR-008**: System MUST allow manual refresh to bypass all caches and fetch fresh data from external services
- **FR-009**: System MUST ensure that the real-time subscription flow does not emit repeated refresh calls when the underlying data has not changed

**Phase 2 — Frontend Refresh Path Fixes**

- **FR-010**: System MUST decouple lightweight task updates (single-issue status changes) from the full board data query invalidation path
- **FR-011**: System MUST trigger a full board data query only on manual refresh or structural board changes (column add/remove, bulk updates)
- **FR-012**: System MUST ensure that real-time sync, fallback polling, auto-refresh, and manual refresh follow a single coherent refresh policy

**Phase 3 — Frontend Render Optimization**

- **FR-013**: System MUST ensure that when a single card's data changes, only the affected card and its parent column re-render
- **FR-014**: System MUST cache derived data (sorting, aggregation, grouping) and recompute only when the underlying board data changes
- **FR-015**: System MUST throttle high-frequency event listeners (drag handlers, popover positioning) to prevent excessive computation during board interactions
- **FR-016**: System MUST stabilize component props where useful to prevent unnecessary re-renders of child components

**Phase 3 — Verification and Regression**

- **FR-017**: System MUST maintain or extend existing automated test coverage for backend cache behavior, change detection, fallback polling, and frontend board refresh logic
- **FR-018**: System MUST complete the before/after checklist with post-optimization measurements demonstrating actual improvements

**Scope Boundaries**

- **FR-019**: System MUST NOT introduce board virtualization, major service decomposition, new external dependencies, or large architectural rewrites in the first pass unless baseline measurements prove they are necessary
- **FR-020**: System MUST NOT remove or weaken existing manual refresh behavior — users must always be able to force a fresh data fetch

### Key Entities

- **Baseline Measurement**: A recorded numeric value for a specific performance metric, with a measurement method, timestamp, and target improvement range. Used to compare before/after optimization results.
- **Board Data Cache**: A time-limited store of previously fetched board data, including sub-issue data. Defined by a validity window (TTL) and invalidation rules (manual refresh bypasses, structural changes invalidate).
- **Refresh Policy**: The rules governing when and how the system fetches fresh board data. Includes four refresh sources (real-time sync, fallback polling, auto-refresh, manual refresh) each with distinct behaviors.
- **Before/After Checklist**: A structured document listing all performance metrics with baseline (pre-optimization) and final (post-optimization) values. Serves as the primary evidence of improvement.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Idle backend API call count for an open board over a five-minute window is reduced by at least 50% compared to the documented baseline
- **SC-002**: Board data endpoint response time shows no regression (remains within 10% of baseline) after optimization changes
- **SC-003**: Frontend board load time-to-interactive shows no regression (remains within 10% of baseline) after optimization changes
- **SC-004**: Unnecessary frontend component re-renders during common board interactions (card drag, card click, status update) are reduced by at least 40% compared to the documented baseline
- **SC-005**: Single-task status updates arriving via real-time sync are reflected in the UI without triggering a full board data refresh
- **SC-006**: Manual refresh continues to bypass all caches and return fresh data in 100% of test cases
- **SC-007**: All existing automated tests (backend cache, board, projects, polling; frontend real-time sync, board refresh) continue to pass with no regressions
- **SC-008**: A completed before/after checklist is produced with numeric baseline and final values for every measured metric
- **SC-009**: At least one manual network and profile pass confirms that real-time updates arrive within 2 seconds, fallback polling does not trigger full refreshes, and board interactions remain responsive
- **SC-010**: A documented second-wave assessment is produced stating either "first pass is sufficient" or recommending specific follow-on work with justification

## Assumptions

- The existing board endpoint already implements a 300-second TTL cache and clears sub-issue cache on manual refresh. The optimization work targets any still-missing pieces rather than reimplementing completed functionality.
- The system already has real-time sync (WebSocket-based) and fallback polling paths. The optimization work refines their behavior rather than replacing the underlying transport.
- Existing automated test coverage for cache, polling, WebSocket fallback, and board refresh behavior is sufficient to serve as regression guardrails and will be extended rather than rewritten.
- Representative board data for measurement purposes is available in the existing development or staging environment without requiring special setup.
- A "large board" for edge-case testing is defined as one with 10+ columns and 100+ cards, based on typical usage patterns.
- Performance improvements are validated against the same hardware and environment used for baseline measurement to ensure fair comparison.

## Dependencies

- Baseline measurement (User Story 1 / Phase 1) blocks all optimization work (User Stories 2–4 / Phases 2–3).
- Backend API fixes (User Story 2) and frontend refresh path fixes (User Story 3) can proceed in parallel once the baseline is established and the desired refresh contract is confirmed.
- Frontend render optimization (User Story 4) can proceed in parallel with backend work but should follow frontend refresh path fixes to avoid conflicting changes.
- Verification (User Story 5) depends on completion of all optimization stories (User Stories 2–4).
- Second-wave assessment (User Story 6) depends on completed verification with documented measurements.

## Out of Scope (First Pass)

- Board virtualization (rendering only visible cards/columns)
- Major service decomposition of the board fetching or polling pipeline
- Introduction of new external dependencies or libraries
- Large architectural rewrites to the real-time sync or caching infrastructure
- Changes to the data model or external API contracts
- Performance optimization of non-board surfaces (unless trivially coupled to board fixes)
